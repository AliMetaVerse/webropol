Typography Sizing — Best Practices

Research-backed guidance from Apple HIG, Google Material Design 3, WCAG 2.1, and typographic theory.

Apple Human Interface Guidelines — Dynamic Type

Use the Dynamic Type scale

Apple defines 11 text styles rather than raw pixel sizes. The body style is 17pt on iOS. Users can scale all text up/down via accessibility settings — your layout must reflow gracefully. Never hard-lock a size that defeats user scaling.

Minimum legible size: 11pt

Apple considers 11pt (~14–15px) the practical floor for interface text. Below this, legibility degrades even on Retina displays. Caption and footnote styles sit at 11–13pt by design.

Line height is proportional, not fixed

SF Pro's built-in metrics set line height at roughly 1.2× for headings and 1.4–1.5× for body. Apple does not publish a universal line-height rule — let the system font handle it unless you have a specific override reason.

Limit type levels per screen

Apple recommends using no more than two or three font sizes in any single view. Hierarchy should come from weight (Regular → Semibold → Bold) and color contrast first; size jumps second.

Google Material Design 3 — Type Scale

Roles, not sizes

Material 3 defines 15 type roles grouped into Display, Headline, Title, Body, and Label. Each role carries a semantic meaning — use Display only for short, expressive hero text; never for body copy.

Body Large = 16sp baseline

Google sets Body Large at 16sp as the reading baseline for paragraph text. sp units scale with Android system font-size preference, equivalent to px at default settings — closely matching CSS rem philosophy.

Minimum body text: 12sp; absolute minimum: 11sp

Label Small (11sp) is the lowest level in the official scale, reserved for compact annotation. Going below 11sp is explicitly not recommended in Material guidelines.

Line height is role-specific

Material 3 publishes exact line-height values per role. Display Large uses 64sp line height on 57sp (ratio ≈ 1.12), while Body Medium uses 24sp on 14sp (ratio ≈ 1.71). Tighter at larger sizes, looser at smaller — the same pattern Webropol follows.

Limit typefaces to two families maximum

Material strongly recommends one primary typeface + one optional secondary. Mixing more than two creates visual noise. Roboto + Roboto Condensed follows this rule precisely.

WCAG 2.1 / W3C — Accessibility

Contrast ratio 4.5:1 for normal text (AA)

All text below 18pt regular (≈24px) or 14pt bold (≈18.67px) must achieve a minimum 4.5:1 contrast ratio against its background. Font size affects the threshold — larger text gets a relaxed 3:1 ratio (AA large).

Text must scale to 200% without loss of content (SC 1.4.4)

Use rem or em units where possible so text respects the browser's base font size. Avoid px for body sizes if you want to pass Resize Text criterion. Fixed-px headings are acceptable if layout reflowing is not needed.

Text Spacing — SC 1.4.12

Content must not lose information when users apply: line-height ≥ 1.5× font size, letter spacing ≥ 0.12×, word spacing ≥ 0.16×, and paragraph spacing ≥ 2×. Build layouts that can absorb these overrides without clipping or overlap.

Never use text smaller than 9px (7pt) anywhere

While WCAG doesn't define a hard minimum size, both Apple and Google set 11–12px as the practical floor. Decorative micro-text under 9px fails basic legibility tests and should be replaced with icons or removed.

Touch target minimum: 44×44pt (Apple) / 48×48dp (Google)

Text that also serves as a tap target (buttons, links) must meet minimum touch target dimensions regardless of font size. A 12px label on a 32px-tall button with no padding fails this criterion.

Typographic Theory — Modular Scale & Layout

Modular scale: consistent ratio between steps

A modular scale multiplies a base size by a fixed ratio at each step. Common ratios: Major Third (×1.25), Perfect Fourth (×1.333), Golden Ratio (×1.618). Webropol's scale approximates a Perfect Fourth — note how H6 18px → H5 20px → H4 23px → H3 26px each step is ≈×1.12–1.26.

Optimal line height: 1.4–1.6× for body text

Research from the Baymard Institute and the Nielsen Norman Group recommends 1.4–1.6× line height for paragraph text. Tighter (1.2×) works for headings where descenders don't collide. Webropol's body sizes use 24/16 = 1.5 and 20/14 ≈ 1.43 — within the optimal range.

Optimal line length: 50–75 characters (45–85 max)

At 16px body text on a standard viewport, this translates to roughly 560–680px column width. Very wide columns (>900px) cause the eye to lose its place between lines. Very narrow (<200px) forces excessive hyphenation. Use max-width on text containers, not the viewport.

Aim for 3–4 distinct hierarchy levels maximum

Using all 6 heading levels plus body and caption in one view creates visual chaos. Most interfaces need only 3 levels: a prominent title, a supporting subheading, and body. Reserve H5–H6 for dense data-heavy screens like dashboards or settings pages.

Desktop-to-mobile size reduction: 15–25%

Both Apple and Material converge on reducing heading sizes by roughly 15–25% for mobile. Webropol's H1 drops from 32px to 26px (−19%), H2 from 29px to 23px (−21%), H3 from 26px to 20px (−23%) — all within the recommended range. Body text may not need to reduce if it is already at 14–16px.

How Webropol aligns with these standards

Heading scale matches Material Headline tokens closely

Webropol H1 (32px/40lh) is identical to Material's Headline Large (32sp/40lh). H2 29px ≈ Material Headline Medium 28sp. The alignment is intentional and ensures cross-platform visual consistency.

Body line height within optimal range

Paragraph Large (16px/24lh = 1.5×) and Paragraph Medium (14px/20lh = 1.43×) both fall in the 1.4–1.6 optimal range defined by NNG and Baymard research.

Mobile reduction is 15–25% — within best-practice range

All heading levels drop between 18–23% from desktop to mobile, matching both Apple's Dynamic Type scaling behaviour and Material's adaptive sizing recommendations.

Watch: Micro paragraph (12px) near WCAG floor

Paragraph Micro at 12px is within the Material Label range and above Apple's 11pt floor — but ensure contrast ratio stays at 4.5:1 or higher when using it at #787F81 grey on white. That pairing passes AA at 12px only if the text remains static (non-interactive).

Suggestion: consider adding rem equivalents in token docs

All sizes are currently published in px. Adding rem equivalents (e.g. H1 = 2rem at 16px base) would make the scale usable in rem-based component libraries and would improve WCAG 1.4.4 (Resize Text) compliance in non-Tailwind contexts.
